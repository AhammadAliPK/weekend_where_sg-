## Key Decisions

### Design Paradigm
- DECISION: Layered architecture with clear separation of concerns
- Frontend (presentation) → API Layer (contracts) → Business Logic (scoring) → Data Layer (sources)
- Enables independent frontend/backend development for training demo

### Stack Choices (Updated for Production)
- DECISION: Node.js 20.x LTS backend with Express 4.x
- DECISION: SvelteJS frontend (modern, reactive, component-based)
- DECISION: shadcn-svelte components (pre-built, accessible, customizable)
- DECISION: Turborepo monorepo (unified build, shared configs, efficient caching)
- DECISION: TypeScript for type safety across frontend and backend
- DECISION: Fallback JSON data for reliability during demo

### Monorepo Structure
- DECISION: apps/web (Svelte frontend)
- DECISION: apps/api (Express backend)
- DECISION: packages/ui (shared shadcn-svelte components)
- DECISION: packages/config (shared ESLint, TypeScript, Tailwind configs)
- DECISION: packages/types (shared TypeScript types)

### Architectural Boundaries
- DECISION: Frontend owns only presentation and API calls
- DECISION: Backend owns all scoring logic and data normalization
- DECISION: API contracts are the only shared truth between frontend/backend
- DECISION: Data sources are pluggable (fallback → live API → future extensions)

### State Management
- DECISION: Svelte stores for frontend state (reactive, simple)
- DECISION: Backend remains stateless (all state in frontend)
- DECISION: No user session management for MVP (single-page experience)

### Error Handling Strategy
- DECISION: Graceful degradation at every layer
- DECISION: Frontend shows user-friendly messages (never technical errors)
- DECISION: Backend returns fallback data when external sources fail
- DECISION: API always returns valid JSON structure (never undefined/null)

### Data Flow
- DECISION: Unidirectional data flow (selectors → API request → response → UI update)
- DECISION: Debounced API calls to prevent excessive requests
- DECISION: Response structure is fixed by API contract (FR-8.2)

### Extensibility Points
- DECISION: Scoring function is pure function (easy to test and modify)
- DECISION: Data sources abstracted behind interface pattern
- DECISION: Signal-based scoring (add new signals without breaking existing ones)
- DECISION: Verdict mapping configurable (separate from scoring logic)
